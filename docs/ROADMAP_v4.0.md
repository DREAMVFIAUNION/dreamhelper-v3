# DREAMVFIA v4.0 路线图 (Roadmap)

> **创建日期:** 2026-03-28  
> **核心目标:** 将系统从功能堆砌推向专业级与拟真级双脑协作，构建真正实用的 Agentization （完全智能体化）极客底座。

---

## 现状简述 (Where are we)
在 v3.x 系列版本中，我们成功实现了**双脑流式融合基础能力**（左右脑并发+皮层仲裁），并且扩展了多达 **100+ 项功能技能树**（图像、音频、通用工具等）。但在实际深度使用中遇到瓶颈：
- **双脑过度并行与资源浪费**：缺乏对任务复杂度的前置鉴别，导致简单任务也在并行烧毁成本。
- **技能上下文爆炸**：静态将 100+ 工具描述注入 Prompt 严重拖慢 LLM 上下文并显著削弱意图命中率。
- **物理环境能力割裂**：尚未完全形成一套安全、可视化的本地化 Agent 工作流。

为此，提出 v4.0 将进行的三项架构革命。

---

## 核心演进支柱 (The Pillars of v4.0)

### 1. DREAMVFIA Fusion 2.0：拟真双脑协作深度进化
不再是简单的“盲写代码，然后提交给裁判”，而是建立基于时间线的同步纠偏机制。

- **动态全脑域平衡比 (Dynamic Hemispheric Weighting)**
  - 根据输入请求的特征集（`intent`, `complexity`, `creativity`, `logic`），**丘脑**先验判断主导脑比例（例如左脑权重 80%：右脑权重 20%）。
  - 小脑拦截机制独立：针对高频快反闲聊（低于 2 Tokens），由小脑低耗模型直接拦截输出，释放主脑算力开销。
- **胼胝体通信与事件总线 (Real-time Corpus Callosum Sync)**
  - 设计局部 Token 缓冲区。流式生成中，一旦右脑检测到左脑输出逻辑冲突超过预设差异阈值，立即触发“辩论模式”。
  - 非最终仲裁，而是实时截断与自我修正。
- **异构多模态并发分工 (Heterogeneous Modality Allocation)**
  - 真正实现“多脑协同”：当用户请求分析一段视频并编写总结的自动化脚本时，右脑专注解析像素池、截帧总结特征；左脑获取右脑透传的语义结构树，自动生成对应的解析与总结 Python 脚本。二人各司其职。

### 2. 技能树重构与语义工具链 (Skill Tree Optimization)
解决上下文窗滥用的系统级别破局。

- **动态语义检索召回 (Semantic Skill Routing)**
  - 将所有技能（包括其调用描述、范例）预存至本地向量存储（Milvus/Elasticsearch）。
  - 在用户流转过程（脑干拆解时），根据当前的 `query` 召回余弦相似度最高的 Top-5 技能进入活跃窗口，而非 100+ 全量加载。
- **全面拥抱 MCP 机制 (MCP Native Integration)**
  - 全面剔除遗留的硬编码工具 `file_read`, `shell_exec`，将全套系统底层能力迁移至纯正的 **Model Context Protocol**。
  - 通过 MCP 服务器隔离不规范调用，统一工具挂载接口协议，开放给第三方开发者标准接入通道。
- **复合工作流封转 (Workflow as a Skill)**
  - 高级用户可以通过 DAG （有向无环图）定义“提取-转换-生成”流，并将其固化为一个新的宏技能挂载回语义池中。

### 3. 极客化本地物理基座 (Local-First Geek Foundation)
围绕 **BYOK (Bring Your Own Key)** 与纯本地运算核心打通。

- **去中心化配置零登录**
  - 不再依赖公网用户鉴权，全量数据存放本地单用户 SQLite 或 PgSQL 隔离层。本地运行、本地存留，守护数据绝对隐私。
- **终端 Xterm 可视化与 Agent 代理执行可视墙**
  - 核心痛点：现在执行 `pnpm install` 或大量脚手架生成属于“静默盲盒执行”。
  - 升级点：前端引入 Xterm 面板。双脑除了处理输出，其中一脑会专职读取终端日志输出。若检测到 `npm ERR!`，立即自主下发 `fix` 工具指令尝试解决，而用户可以通过面板实时察看 Agent 的一切挽救行为。

  - 实现双级错误自修复逻辑。与前端 GUI 控制台完全统合。
